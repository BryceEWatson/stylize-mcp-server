## Building Stylize: Crafting a Cloud-Native Image Orchestration Service on GCP

In the rapidly evolving world of generative AI and developer tools, the demand for seamless, integrated image stylization is booming. That's the core idea behind **Stylize MCP Server**: a service designed to be the go-to style orchestrator for any Model-Context-Protocol (MCP) aware agent, transforming user images into high-quality, style-specific variants with a single call.

This isn't just another image filter. Stylize aims to solve real pain points for developers, designers, and even automated marketing agents by bundling complex concerns—multi-engine orchestration (starting with DALL·E 3), curated style banks, prompt engineering, content safety, caching, and cost controls—into one frictionless MCP tool.

As we embark on building the MVP, laying a robust, scalable, and *cost-conscious* foundation on Google Cloud Platform is critical. This post shares our journey through the initial infrastructure provisioning using Terraform, highlighting the technical decisions and rationale that an engineering audience might find interesting.

**The Vision: More Than Just an API Wrapper**

Before diving into the "how," let's touch on the "why." Existing MCP image tools are often basic, requiring users to manage raw API calls, prompt engineering, and crucially, costs and safety. Stylize differentiates by offering:

*   A **curated style catalog** (think Vaporwave, Zen-Ink, but also UI elements & logos).
*   **Intelligent prompt templating**.
*   **Built-in safety filtering** via Cloud Vision API.
*   **Aggressive caching** and **cost guardrails** (essential when dealing with per-image API costs).
*   A **cloud-native architecture** designed for scalability on GCP.

Our goal is to become a staple in MCP marketplaces like Cline, Windsurf, and Copilot Studio.

**Architecting for the Future, Building for the MVP on GCP**

Our high-level architecture (see Vision Document Section 6) leverages Cloud Run as the stateless compute core, Firestore for metadata, Memorystore (Redis) for caching, Pub/Sub for async tasks (like future multi-engine calls to DALL·E 3 and Vertex AI Imagen), and Cloud Storage for images.

For the MVP development phase, however, **cost optimization is key**. We're taking a pragmatic approach:

1.  **Core Infrastructure with Terraform:** We've defined:
    *   A custom **VPC Network** and initial subnet.
    *   A dedicated **Service Account (`stylize-mcp-sa`)** with granular IAM permissions:
        *   `roles/storage.objectAdmin` (GCS)
        *   `roles/datastore.user` (Firestore)
        *   `roles/aiplatform.user` (Vision API - this was a learning; `roles/cloudvision.user` initially failed at the project level)
        *   `roles/pubsub.publisher` (Analytics)
        *   `roles/secretmanager.secretAccessor` (OpenAI API Key)
    *   **GCS Buckets** for original and stylized images, with lifecycle rules.
    *   A **Pub/Sub Topic** for future analytics.
    *   **Secret Manager** for our `OPENAI_API_KEY` (using a placeholder in Terraform, manual update for security).
    *   Enabled all necessary **GCP APIs** (Cloud Run, Build, Vision, etc., including `compute.googleapis.com` which was an early "gotcha" for VPC creation).

2.  **The "Lean Dev" Cost Strategy:**
    *   **Deferred Big Ticket Items:** For active development, we've temporarily **commented out Memorystore for Redis, Serverless VPC Access Connector, and Cloud NAT** from our Terraform configuration. This dramatically reduces our immediate idle costs (saving an estimated ~$85/month).
    *   **Impact & Mitigation:** While this means no server-side caching and default internet egress for Cloud Run in our dev deployment, it's acceptable for the initial feature-building sprints. Caching logic will be developed locally with Docker. These services will be re-introduced as we approach testing for those specific features or prepare for wider use. Our application code will be designed to handle the absence of these services gracefully.

3.  **Serverless First with Cloud Run:**
    *   Our application (FastAPI + FastMCP) will be containerized and deployed to **Cloud Run**.
    *   We've defined the `google_cloud_run_v2_service` in Terraform with a placeholder image, essential environment variables (like the path to the OpenAI key in Secret Manager), and the `stylize-mcp-sa` service account.
    *   Crucially, `lifecycle { ignore_changes = [template.0.containers.0.image] }` is set, allowing our CI/CD pipeline to manage image updates.
    *   For MVP, it's configured for public access.

4.  **Automated Deployments with Cloud Build:**
    *   A `Dockerfile` standardizes our Python environment.
    *   `cloudbuild.yaml` orchestrates building the image, pushing it to Artifact Registry, and deploying to Cloud Run. It uses substitution variables for dynamic configuration (region, service names, and critical env vars like the OpenAI secret path).
    *   We've documented the manual steps for setting up the Artifact Registry repo and the Cloud Build trigger, including the vital IAM permissions for the Cloud Build service account (like `roles/run.admin`, `roles/artifactregistry.writer`, `roles/iam.serviceAccountUser`).

**Navigating the Initial `terraform apply`:**

The first full `terraform apply` is always an adventure. We hit a few common bumps:
*   **Local Authentication:** Ensured both `gcloud auth login` and `gcloud auth application-default login` were run.
*   **API Enablement:** Explicitly adding `compute.googleapis.com` to our API enablement list was necessary for VPC creation. We also noted that API enablement can have propagation delays.
*   **IAM Role Specificity:** The `roles/cloudvision.user` didn't apply at the project level as expected. Switching to the slightly broader but commonly used `roles/aiplatform.user` resolved the permission issue for our service account to call the Vision API.

**Current Status & What's Next**

With these infrastructure pieces (and cost optimizations) in place and successfully provisioned by Terraform, our `stylize-mcp-server` is live on Cloud Run with a placeholder application. The `OPENAI_API_KEY` is securely set, and our Cloud Build CI/CD pipeline has successfully deployed the initial container.

This solid, automated, and cost-aware foundation means we can now confidently shift focus to the core application development: implementing the `stylize_image` endpoint, integrating DALL·E 3, building out the style catalog, and implementing the SafeSearch and quota features that will make Stylize a valuable tool for its users.

Building robust cloud infrastructure is an iterative process. By starting lean, automating early, and being prepared to troubleshoot, we're setting Stylize up for success in the growing MCP ecosystem.
