**Stylize MCP Server: From Infrastructure-as-Code to an Operational CI/CD Pipeline on GCP**

Following our initial infrastructure layout for the Stylize MCP Server MVP on Google Cloud Platform (as detailed in our previous post), today's efforts centered on the critical "last mile": operationalizing our Continuous Integration/Continuous Deployment (CI/CD) pipeline and successfully deploying our initial application container. This phase, while detailed, is foundational for rapid development and iteration.

**Finalizing GCP Configuration via CLI**

With our Terraform baseline applied, several crucial one-time configurations were executed using `gcloud` CLI, guided by our `MANUAL_SETUP_CHECKLIST.md`. This ensured reproducibility and supplemented our IaC approach for elements like service connections and trigger definitions:

1.  **Secure API Key Management:** The placeholder `OPENAI_API_KEY` in GCP Secret Manager was updated with the operational key.
2.  **Artifact Registry:** The `stylize-repo` Docker repository was provisioned in `us-central1` to host our container images.
3.  **Cloud Build & GitHub Integration:**
    *   A **Cloud Build GitHub Connection** (`stylize-github-connection`) was established. This involved an initial permission fix: granting the Cloud Build service agent (`service-[PROJECT_NUMBER]@gcp-sa-cloudbuild.iam.gserviceaccount.com`) `roles/secretmanager.admin` to manage its internal secrets for the connection. User OAuth authorization completed this step.
    *   Our specific repository (`BryceEWatson/stylize-mcp-server`) was then explicitly linked to this connection via the GCP Console, a necessary step before trigger creation.
4.  **Cloud Build Trigger Automation:** After significant iteration to match GCP's expected syntax for global triggers tied to specific service accounts and GitHub App integrations (as opposed to regional triggers or newer Connection API patterns for the trigger definition itself), the following `gcloud` command successfully created our primary deployment trigger:
    ```bash
    gcloud builds triggers create github \
      --name=stylize-mcp-server-deploy \
      --project=stylize-mcp-server \
      --region=global \
      --repo-owner=BryceEWatson \
      --repo-name=stylize-mcp-server \
      --branch-pattern="^main$" \
      --build-config=cloudbuild.yaml \
      --service-account=projects/stylize-mcp-server/serviceAccounts/stylize-mcp-sa@stylize-mcp-server.iam.gserviceaccount.com \
      --substitutions='_REGION=us-central1,_ARTIFACT_REPO_NAME=stylize-repo,_SERVICE_NAME=stylize-mcp-server,_OPENAI_API_KEY_SECRET_PATH=projects/stylize-mcp-server/secrets/OPENAI_API_KEY/versions/latest'
    ```
    *Key learning: Explicitly setting `--region=global`, specifying the `--service-account` for the trigger, and ensuring substitutions were correctly parsed (using single quotes around the entire list for PowerShell) were vital.* We also confirmed correct substitution variable setup using `gcloud builds triggers describe ... --format=json` after an update.
5.  **Billing Oversight:** A budget alert for our project was configured via `gcloud beta billing budgets create ...`, including enabling the `billingbudgets.googleapis.com` API.

**Iterative Refinement of CI/CD Build Permissions**

Our first automated build, triggered by a test commit, surfaced a series of IAM permission requirements for the custom service account (`stylize-mcp-sa@...`) that our trigger uses to execute build steps:

*   **Initial Error (Logging Configuration):** Build failed because a custom service account requires explicit log bucket configuration or specific logging options.
    *   **Resolution:** Added `options: { logging: CLOUD_LOGGING_ONLY }` to `cloudbuild.yaml`.
*   **Subsequent IAM Errors & Fixes (for `stylize-mcp-sa`):**
    1.  `roles/logging.logWriter`: Granted to allow writing build logs to Cloud Logging.
    2.  `roles/artifactregistry.writer`: Granted to allow pushing the built Docker image to Artifact Registry.
    3.  `roles/run.admin`: Granted to allow deploying and managing the Cloud Run service.
    4.  `roles/iam.serviceAccountUser` (on `stylize-mcp-sa` itself): Granted to allow the service account to deploy a Cloud Run service that runs *as itself*.

These permissions were primarily added to our `infra/iam.tf` file and applied via `terraform apply` to maintain IaC principles, with `gcloud` commands used for immediate testing where necessary.
